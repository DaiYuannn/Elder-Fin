# Elder-Fin 项目任务路径

## 固定工作规则

1. 每次进行项目制作或代码修改之前，必须先阅读本文件。
2. 每次完成项目制作或代码修改之后，必须再次阅读本文件，确认当前改动是否仍然符合既定目标。
3. 本文件作为当前项目的统一任务路径，不允许在未阅读的前提下直接修改页面、样式或脚本。
4. 后续如果任务目标发生变化，先更新本文件，再开始改代码。

## 当前项目总目标

项目名称：Elder-Fin 老年财经伴侣中枢平台

目标分为两条主线：

- C 端：面向老年用户和家属，优先手机端适配，强调大字、易读、易操作、风险提示直白，并预留语音功能接口。
- B 端：面向后端处置人员，优先电脑端适配，强调信息密度、模块化工作台、地图态势、风险事件联动和快速处置能力，并进入独立 SPA 指挥中枢路线。

## 当前已确定页面策略

- 首页保留展示页定位，用于说明项目价值、能力和比赛展示。
- 首页右上角使用登录入口，不再直接放 B 端入口按钮。
- 登录后按身份跳转不同界面。
- 项目进入全量 React 化重构阶段，根目录建立统一 React 应用承接首页、登录页、老人端、家属端与后续统一路由。
- B 端继续保留联合干预中枢路线，但从“独立孤立应用”调整为“优先复用现有 biz-console 结构并逐步并入统一 React 主工程”的策略。

## 当前角色分流目标

- 老年用户：进入老人端主界面。
- 家属用户：进入家属看板或家属协同界面。
- B 端用户：进入 Elder-Fin 联合干预中枢。

## 当前制作顺序

1. 先在根目录搭建统一 React + Vite 主工程，并接管首页与登录分流。
2. 再迁移老人端与家属端页面，拆出共享路由、角色守卫、状态层与本地存储服务。
3. 同步梳理现有静态脚本中的数据模型，把风险记录、家属联系人、角色信息统一抽离。
4. B 端继续沿用现有四大模块方向，优先复用 biz-console 中已完成的地图、布局和 Zustand 状态设计。
5. 在统一 React 工程中逐步合并 B 端入口，淘汰旧的静态 admin / risk / family 三页后台演示结构。
6. 最后再接实时数据流、图层联动、更细粒度区划与真实接口。

## 当前 B 端目标

- 页面名称统一为 Elder-Fin 联合干预中枢。
- B 端当前已有 biz-console 目录，技术栈保持 React + Vite + Tailwind CSS + react-grid-layout + Zustand + deck.gl + MapLibre GL JS，不推倒重来。
- B 端不再分裂成总览页、风险页、家庭页三套静态页面，而是改为一个全局指挥画面。
- 全局画面固定四大模块：MapStage（中上）、OverallInfo（左上）、FamilyFlow（左下）、RiskStream（右侧）。
- 四大模块必须支持拖拽、缩放、布局保存和后续扩展。
- MapStage 必须支持国际层、中国省级层、城市层、网点/交易层的语义缩放切换。
- 国际层点位语义为国家；国内中层点位语义为省份或城市；更细层级语义为具体交易坐标或网点。
- 地图层渲染由 deck.gl 承担，底图与缩放控制由 MapLibre 承担。
- 全局 zoom、当前选中事件、模块布局和联动状态由 Zustand 统一管理。
- 实时数据流先采用 mock 数据，后续再接 WebSocket 或 SSE。
- 当前阶段目标：先完成根目录 React 主工程与公共路由骨架，再逐步把现有 biz-console 能力并入统一体系，替换旧 B 端静态演示页。

## 当前 B 端界面说明规则

- 所有“情况说明”“场景解释”“实现说明”类文案，不再直接展示在地图主舞台界面中。
- 界面中只保留操作必需信息：层级状态、缩放值、事件状态、数据源状态、图层开关、风险图例。
- 关于全球视图、省级视图、图层含义、联动策略、时间线说明等解释性内容，统一维护在本任务路径文件中，不再作为 UI 卡片或地图底部说明条展示。
- MapStage 内允许保留轻量 hover 信息和风险图例，但不得再增加长段解释文案或底部说明横条。

## 当前 B 端情况说明归档

- 国际层：使用真实国界 GeoJSON、国际端点和跨境飞线，强调跨境资金风险路径。
- 省级层：使用中国省级边界、目标省份高亮和省际链路，强调国内区域联动。
- 城市层：使用城市节点与城市间链路，作为省级以下的中间态势层。
- 网点/交易层：使用更细粒度网点或交易坐标，承接最终处置定位。
- 风险图例：颜色从蓝到黄再到红，表示从低风险到高风险的热度渐变。
- 标签策略：地图中优先展示当前激活事件节点，其余标签按层级裁剪，避免信息过载。
- hover 信息：仅保留事件编号、链路、金额与状态，作为轻量提示，不承担说明文档功能。

## 当前 C 端目标

- 老年端在 React 化后继续保持适老化风格，不牺牲大字、直白提示和少步骤交互。
- 老年用户登录后优先进入老人端主界面，由首页统一汇总风险提醒、家属协同和下一步操作入口。
- 老人端主界面继续强化“今日守护状态”、快捷入口和手机端单手操作体验，优先把高频动作做成大按钮。
- 家属端继续围绕手机查看场景优化，优先突出摘要、联系人和待外呼事项，不把桌面表格当成唯一信息形态。
- 家属端的外呼记录、确认结果与后续备注进入统一状态层维护，避免只停留在一次性页面展示。
- 老人端与家属端在移动端统一使用底部快捷导航，缩短单手操作路径。
- 移动端优先，并在组件层面保留后续语音入口和方言 AI 接口位置。
- 家属端与老人端共享统一前端底座，但视觉与交互节奏保持独立，不向 B 端高密度工作台靠拢。

## 当前统一前端重构要求

- 根目录不再继续扩展原有 HTML + assets/app.js 的页面逻辑，新功能默认落在 React 主工程。
- React 主工程需优先建立：路由系统、角色分流、角色守卫、共享布局、本地存储服务、页面级状态拆分。
- 旧静态页面在迁移完成前可作为对照参考，但不再作为长期主线继续演化。
- 首页、登录页、老人端、家属端应统一纳入一个主应用中；B 端在稳定后并入同一套路由体系。
- 所有后续页面改造必须优先考虑状态统一，避免重复维护两套事件流、联系人流和登录流。
- 对外演示优先支持 GitHub Pages 这类纯静态托管方式，打包产物不得依赖开发服务器路径。

## 当前模拟数据规则

- 老人端、家属看板和 B 端本地模式统一共享同一份模拟 alerts 与 contacts 数据，不允许三端各自维护独立样例。
- 模拟数据优先放在统一状态层中维护，由 C 端和家属端直接展示，由 B 端通过 unifiedConsoleBridge 映射为中枢事件流。
- 家属端新增的外呼记录、确认结果和备注，在 B 端本地模式下也要同步映射到事件详情与家庭协作信息中。
- 如果后续继续补模拟场景，优先增加共享事件、联系人和派生展示，不单独在某个页面里硬编码一套无法同步到其他端的假数据。

## 已迁出 UI 的制作说明归档

- 首页曾展示“统一 React 主工程”“当前迁移策略”“后续并入统一 React 主工程”等制作说明，这些内容不再直接面向用户展示。
- 登录页曾展示“当前已切入 React 主工程”“后续再接账号、短信验证码和权限体系”“当前先保留迁移入口”等研发阶段文案，这些内容统一迁移到本文件维护。
- 跨身份访问提示页曾展示“此时界面应该显示什么”“当前策略是不再直接打回登录页”等实现说明，现已改为纯权限提示文案。
- 家属授权页曾展示“数据仍暂存于浏览器本地”等实现细节，现已从界面移除。
- 家属看板页脚曾展示“已迁入 React 主工程”等迁移状态说明，现已改为纯业务说明。
- B 端中枢页头曾展示“当前已并入根目录统一 React 主工程”“Phase 2 / Unified Route”等工程阶段信息，现已从界面迁出。
- B 端总风险流空态曾展示“后续这里将接入时间轴、联动详情、人工干预动作和实时推送入口”等未来实现提示，现已改为简单操作提示。
- 旧参考目录 biz-console 中对应的 B 端页头和风险流空态提示文案也已同步清理，避免后续复用旧源码时再次带回制作说明。
